### 1.1 编写目的

本文件用于向甲方清晰说明科研成果管理系统下一版本的三项正式交付能力，避免在沟通过程中出现以下偏差：

- 把规划能力误解为已经完全上线的成熟能力。
- 把“辅助识别、辅助判断”误解为“自动决策、自动转化”。
- 把三项能力理解为分散功能点，而非一组面向成果转化与管理辅助的整体增强。

### 1.2 核心结论

本次科研成果管理系统下一版本，建议按以下统一口径与甲方对齐：

- `需求洞察` 是下一版本的主线能力，直接服务成果转化线索发现与成果匹配。
- `研究洞察` 是下一版本的增值能力，直接服务管理层方向研判与项目征集辅助。
- `成果物自动补全` 是下一版本的提效能力，直接服务成果录入效率提升。

可对外统一表达为：

**下一版本将正式交付需求洞察、研究洞察、成果物自动补全三项能力；其中需求洞察主线推进，研究洞察体现管理增值，成果物自动补全体现录入提效。**

## 2. 背景与现状

### 2.1 项目背景

科研成果管理系统当前已经具备成果录入、审核、检索、统计和成果展示等基础能力，能够支撑成果归集与日常管理。但随着甲方对“成果转化率”和“决策辅助能力”的关注增强，现有系统还存在三类明显诉求：

- 科研人员希望减少重复录入，提高成果创建效率。
- 科研管理人员希望更主动地发现外部需求，并与内部成果建立关联。
- 管理层希望从内部成果与外部线索中提炼出更稳定的热点方向与机会方向。

因此，下一版本的增强目标不是简单增加几个页面，而是让系统从“成果管理平台”进一步升级为“成果管理 + 成果转化辅助 + 方向洞察辅助”的平台。

### 2.2 当前真实现状

结合现有仓库与资料，当前系统与本次三项能力相关的真实现状如下：

#### 2.2.1 已具备的基础

- 系统已有完整的成果录入、展示、审核和检索主流程。
- 前端已存在 `需求洞察` 页面承载基础。
- 前端已存在 `研究洞察` 页面承载基础。
- 前端已存在 `数据源管理` 相关页面承载基础。

#### 2.2.2 已真实落地的智能能力

当前已真实接通的成果录入智能能力较少，主要仍停留在局部字段补全层面，例如成果录入助手中的局部补全能力。也就是说，系统已经具备“小范围真实能力”，但尚未形成覆盖论文、软著等常见成果附件的完整识别与预填闭环。

#### 2.2.3 当前差距

当前 `需求洞察`、`研究洞察` 及相关数据源能力，整体仍处于以下阶段：

- 页面原型已具备基本承载形式。
- Mock 或演示数据已能表达产品方向。
- 真实采集、结构化处理、匹配与洞察闭环仍待完成。

因此，本次与甲方对齐时应明确：

**这三项能力均作为下一版本正式交付内容推进，但其成熟度基础不同。当前已有原型基础，下一版本完成正式交付与真实闭环建设。**

### 2.3 本次对齐关注点

本次科研成果管理系统增进点对齐，重点围绕以下三项能力展开：

1. `成果物自动补全`
2. `需求洞察`
3. `研究洞察`

三项能力将在下一版本统一交付，但在业务定位上存在明显差异：

- `需求洞察` 解决成果转化主线问题。
- `研究洞察` 解决管理层方向研判问题。
- `成果物自动补全` 解决录入效率问题。

## 3. 三项增进点说明

### 3.1 增进点一：成果物自动补全

#### 3.1.1 定位

成果物自动补全定位为成果录入环节的提效能力。核心目标不是替代用户创建成果，而是在用户上传论文、软著等附件后，辅助识别关键信息并预填到录入表单中。

#### 3.1.2 建议能力表达

建议对甲方按以下方式表述：

- 用户上传附件后，系统辅助识别标题、作者、单位、摘要、关键词、年份、登记号等关键信息。
- 系统将识别结果以“预填字段 + 原文片段 + 待确认项”的方式呈现。
- 用户确认后再写入正式表单，最终提交仍由人工控制。

#### 3.1.3 下一版本交付范围

下一版本建议正式交付以下内容：

- 支持 PDF、DOC、DOCX 等常见格式的基础识别。
- 优先覆盖论文、软著两类高频成果附件。
- 输出可回填的核心字段、原文依据和待确认项。
- 与现有成果创建流程衔接，不改变主流程职责分工。

#### 3.1.4 验收关注点

建议甲方按以下方向理解验收：

- 上传后是否能稳定生成核心字段预填结果。
- 识别结果是否带有原文依据，便于人工确认。
- 是否明显降低科研人员重复录入工作量。

#### 3.1.5 边界说明

本能力不建议按以下口径对外承诺：

- 不承诺所有附件格式都能达到相同识别效果。
- 不承诺所有字段都能一次性自动填准。
- 不承诺识别结果可以替代人工校对和最终提交。

### 3.2 增进点二：需求洞察

#### 3.2.1 定位

需求洞察定位为面向成果转化场景的主线能力。其核心不是“做一个爬虫页面”，而是建设一条围绕成果转化的业务闭环：

**需求采集 -> 需求入池 -> 摘要与标签 -> 成果匹配 -> 人工确认 -> 跟进沉淀**

#### 3.2.2 建议能力表达

建议对甲方按以下方式表述：

- 系统面向指定白名单企业、机构、政府或公开平台，持续采集与科研合作、技术征集、招投标相关的真实需求线索。
- 系统对需求内容进行摘要、标签和结构化处理。
- 系统将结构化需求与机构已有成果进行匹配，辅助管理员发现潜在转化机会。
- 是否有效、是否跟进、是否进入转化工作，仍由人工确认和推进。

#### 3.2.3 下一版本交付范围

下一版本建议正式交付以下内容：

- 白名单数据源采集与基础调度能力。
- 需求池入库、摘要、关键词和标签结构化能力。
- 需求与成果的相似度匹配和候选结果展示能力。
- 跟进状态记录与人工确认入口。

#### 3.2.4 主线价值

该能力与“提高成果转化率”最直接相关，主要价值在于：

- 让成果库从被动存档转向主动发现机会。
- 减少人工分散搜集需求的重复工作。
- 把需求发现、成果匹配和跟进沉淀到系统内，形成可追溯闭环。

#### 3.2.5 边界说明

本能力不建议对外表述为：

- 全网需求自动发现平台。
- 自动生成转化决策的平台。
- 无需人工介入的自动匹配与推进系统。

### 3.3 增进点三：研究洞察

#### 3.3.1 定位

研究洞察定位为面向管理层和科研管理负责人的增值能力。其核心目标不是处理单条需求，而是帮助回答以下问题：

- 当前哪些研究主题值得重点关注。
- 哪些方向正在升温。
- 机构内部已有成果积累集中在哪些方向。
- 哪些方向更适合后续组织项目征集或资源投入。

#### 3.3.2 建议能力表达

建议对甲方按以下方式表述：

- 系统围绕内部成果、外部需求和主题关键词，形成热点主题、趋势变化、强势方向和机会缺口等管理视图。
- 研究洞察服务于管理判断，不替代管理决策。
- 图谱、词云、关系网络只是展示形式，重点是输出可读的方向结论。

#### 3.3.3 下一版本交付范围

下一版本建议正式交付以下内容：

- 基于内部成果与外部需求的主题归类与热点呈现。
- 主题热度、方向变化和重点关键词汇总视图。
- 内部成果强势方向与外部热点方向的对照展示。
- 面向管理层的可读化结论摘要。

#### 3.3.4 增值价值

该能力的核心价值在于：

- 帮助管理层更快看到值得关注的研究方向。
- 为项目征集、资源配置和对外合作提供辅助依据。
- 让系统从“成果库”进一步延伸为“方向研判辅助工具”。

#### 3.3.5 边界说明

本能力不建议对外表述为：

- 自动给出项目立项结论。
- 自动给出资源配置结论。
- 仅靠词云或图谱就形成完整管理价值。

## 4. 下一版本交付安排

建议下一版本按“三项统一交付、主次清晰、边界明确”的方式与甲方对齐。

| 能力 | 业务定位 | 下一版本交付重点 | 成熟度说明 |
| :--- | :--- | :--- | :--- |
| 需求洞察 | 主线能力 | 需求采集、需求池、结构化处理、成果匹配、跟进闭环 | 已有页面与概念基础，下一版本完成真实闭环交付 |
| 研究洞察 | 增值能力 | 热点主题、趋势变化、方向摘要、供需对照视图 | 已有原型基础，下一版本形成正式管理视图 |
| 成果物自动补全 | 提效能力 | 附件识别、字段预填、原文依据、人工确认回填 | 已有局部能力基础，下一版本形成正式录入提效能力 |

这一安排有两个核心好处：

- 对甲方表达清晰，避免误解为只有其中一项会进入下一版本。
- 对项目实施可控，在同一版本内明确主线、增值和提效三类目标。

## 5. 下一版本交付范围与不做内容

### 5.1 下一版本明确纳入

下一版本建议明确纳入以下三项能力：

- `需求洞察`
- `研究洞察`
- `成果物自动补全`

### 5.2 下一版本明确不做

下一版本不建议承诺以下内容：

- 不做全网泛化抓取。
- 不做自动审批、自动决策、自动转化。
- 不做脱离人工确认的成果自动入库。
- 不把研究洞察简化为单纯词云展示页。
- 不承诺所有文档类型和所有字段在首版一次性全覆盖。

### 5.3 后续可扩展方向

在下一版本完成正式交付后，后续可持续增强以下内容：

- 更多成果类型和字段模板的识别能力。
- 更多白名单数据源与更细粒度的数据治理能力。
- 更深入的主题聚类、趋势比较与标杆分析能力。

## 6. 甲方收益

### 6.1 对科研人员

- 成果录入更轻量，重复填写明显减少。
- 上传附件后即可看到待确认的核心字段，降低整理成本。

### 6.2 对科研管理员与转化人员

- 外部需求线索进入系统更持续、更规范。
- 需求与成果的匹配不再只依赖线下经验判断。
- 转化线索发现、确认和跟进形成系统闭环。

### 6.3 对管理层

- 能从系统中看到更清晰的研究热点与方向变化。
- 能更好支持项目征集、方向研判和资源配置讨论。

---

## 7. 验收口径

建议下一版本按以下口径与甲方确认验收：

### 7.1 成果物自动补全

- 上传论文、软著等高频附件后，系统能辅助识别核心字段。
- 识别结果带有原文依据和待确认项。
- 用户确认后可回填至成果创建表单。

### 7.2 需求洞察

- 系统可按白名单数据源采集并形成需求池。
- 需求可生成摘要、标签和结构化结果。
- 需求与成果之间可形成候选匹配结果，并支持人工确认与跟进。

### 7.3 研究洞察

- 系统可基于内部成果与外部需求形成热点主题和趋势视图。
- 可输出面向管理层阅读的方向摘要。
- 支持从主题热点回看相关成果或相关需求依据。
