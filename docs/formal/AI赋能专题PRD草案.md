### 1.1 编写目的

本文件用于对齐科研成果管理系统在 AI 赋能方向上的建设思路、一期落地范围、二期扩展规划及业务价值口径，便于甲方在项目迭代阶段快速理解以下问题：

- 当前系统在 AI 相关能力上已经具备哪些基础。
- 现有能力中哪些是真实落地，哪些仍处于前端原型或 mock 阶段。
- 一期应优先打通哪条业务闭环，才能在有限范围内形成可上线、可验收的成果。
- 二期如何在一期基础上继续扩展，形成更完整的科研管理 AI 助手能力。

### 1.2 文档结论

本次 AI 专题建设建议采用“需求匹配主线，研究洞察增值”的方案路径：

- 一期以“需求洞察与智能匹配”作为主线，形成“需求采集 -> 智能摘要/标签 -> 成果匹配 -> 人工确认/跟进”的最小可用闭环。
- 二期在一期数据资产和运营闭环基础上，增强“研究热点洞察”和“成果录入助手”两类能力。
- 所有 AI 能力均定位为辅助决策能力，而非全自动决策能力，最终判断权仍归业务人员。

## 2. 项目背景与现状

### 2.1 项目背景

科研成果管理系统已经具备成果管理、审核、检索、统计、过程成果对接等基础能力，可以支撑科研成果的归集、审核和展示。但随着业务向成果转化、需求对接、管理决策支持方向延伸，传统规则化能力已难以满足以下场景：

- 外部需求信息分散，依赖人工收集、整理、甄别，效率较低。
- 科研成果与企业/产业需求之间缺少高质量匹配链路，成果转化线索形成慢。
- 管理层缺少基于成果关键词、需求热点、研究方向演化的辅助洞察工具。
- 成果录入过程仍存在元数据补全不足、人工填写负担重的问题。

因此，系统需要引入一套“以 AI 为增强层、以业务闭环为核心”的迭代方案，在不改变现有成果管理主链路的前提下，提升成果利用效率和管理决策质量。

### 2.2 当前系统 AI 相关现状

基于当前仓库梳理，AI 相关能力可分为三类现状：

#### 2.2.1 已有前端方案壳

系统前端已经具备若干 AI 赋能模块的页面原型和交互雏形：

- `需求洞察与智能匹配`
  - 页面已具备需求列表、需求详情、智能摘要、匹配结果、重匹配等交互结构。
- `研究洞察`
  - 页面已具备关键词图谱、时间范围筛选、热点可视化展示等交互结构。
- `系统设置 / 爬虫数据源`
  - 页面已具备数据源配置、抓取频率、去重阈值、通知策略等管理能力设计。

#### 2.2.2 已有真实后端能力

当前仓库中真实落地的 AI 相关后端能力较少，已明确落地的仅有：

- `成果录入助手中的期刊等级补全`
  - 后端通过 `/results/auto-fill` 提供期刊等级查询能力。
  - 当前实际支持类型为 `journalRank`，通过 EasyScholar 代理调用。

#### 2.2.3 原型与真实闭环之间的差距

当前“需求洞察”“研究洞察”“爬虫数据源管理”等能力主要停留在前端方案和 mock 层，尚未形成真实后端数据闭环。因此，当前 AI 模块的整体状态应定义为：

**部分原型已成形，真实闭环待打通。**

本次专题 PRD 的出发点不是把现有原型直接包装成“已上线能力”，而是基于真实现状，规划一条可上线、可验收、可扩展的建设路径。

## 3. 目标用户与核心痛点

### 3.1 目标用户

本专题主要面向高校/科研院所场景，重点涉及以下角色：

| 角色 | 核心关注点 |
| :--- | :--- |
| 科研管理员 | 希望快速掌握外部需求、提高需求筛选效率、促进成果与需求的对接落地 |
| 科研人员 | 希望便捷发现潜在合作需求，降低成果录入负担，提高成果曝光与转化机会 |
| 管理层 / 决策者 | 希望掌握研究热点、需求趋势、成果转化机会，辅助资源配置与方向判断 |

### 3.2 甲方核心痛点

结合当前系统现状和高校/科研院所业务特点，AI 专题优先解决以下三项痛点：

#### 痛点一：外部需求信息分散，人工收集与筛选效率低

- 企业需求、园区需求、政府采购需求、合作需求分布在多个来源。
- 当前缺少统一采集、分类、去重和可信度评估机制。
- 管理员需要花费大量时间做重复的信息整理工作。

#### 痛点二：成果与需求之间缺少高质量匹配链路

- 现有系统已具备成果库，但尚未形成“以需求为入口”的智能匹配能力。
- 需求与成果之间主要依赖人工经验判断，匹配效率和一致性较低。
- 缺少“匹配分数、匹配理由、证据片段、人工重匹配”的标准化流程。

#### 痛点三：管理层缺少对热点趋势与转化机会的直观洞察

- 当前统计能力以成果数量、类型分布等基础统计为主。
- 缺少围绕关键词热度、交叉方向、需求热点、成果转化机会的分析视角。
- 管理层难以基于系统数据快速判断“哪些方向值得关注、哪些需求值得推动”。

## 4. AI 建设目标

### 4.1 总体目标

在不破坏现有成果管理主流程的前提下，为系统增加一层面向成果转化和管理决策的 AI 增强能力，使系统从“成果归档平台”升级为“成果管理与转化辅助平台”。

### 4.2 一期目标

一期目标聚焦于形成一条最小可用业务闭环：

**需求采集 -> 智能摘要/标签 -> 成果匹配 -> 人工确认/跟进**

一期强调以下原则：

- 范围收敛，优先打通一条可上线的业务链路。
- 强调辅助推荐，不替代人工判断。
- 以可落地、可验收为优先，不追求一次性覆盖所有 AI 场景。

### 4.3 二期目标

二期在一期基础上扩展两类能力：

- `研究热点洞察`
  - 面向管理层提供关键词热度、共现关系、时间区间趋势分析。
- `成果录入助手`
  - 在现有期刊等级补全基础上，扩展 DOI、标题、来源元数据补全能力，提升录入效率和数据完整度。

## 5. 建设原则与产品定位

### 5.1 产品定位

本专题中的 AI 能力定位为：

**以成果转化为核心目标、以管理洞察为增值能力、以人工确认为最终闭环的科研管理 AI 助手。**

### 5.2 建设原则

- 以业务闭环为核心，不以模型堆砌为目标。
- 以辅助决策为定位，不以自动决策为定位。
- 先做真实可用，再做高级扩展。
- 明确区分已实现能力与规划能力，避免认知偏差。
- 在可解释性、可追溯性、人工可干预方面优先于“完全自动化”。

## 6. 一期功能方案

一期主线为“需求洞察与智能匹配”，围绕需求侧数据进入系统后的全链路展开。

### 6.1 功能模块一：需求采集管理

#### 6.1.1 目标

将分散在外部渠道的需求信息统一纳入系统，形成标准化、可管理、可追踪的需求池。

#### 6.1.2 功能点

- 支持配置多来源需求数据源。
- 支持配置抓取频率、启停状态、优先级和认证方式。
- 支持对抓取结果进行基础去重与质量状态标记。
- 支持记录数据源运行状态、最近成功时间、失败原因。

#### 6.1.3 业务价值

- 解决需求分散、人工采集成本高的问题。
- 为后续摘要、标签、匹配提供统一数据入口。
- 建立需求资产池，为成果转化工作提供持续线索输入。

### 6.2 功能模块二：需求结构化处理

#### 6.2.1 目标

把原始需求文本转化为可理解、可筛选、可匹配的结构化信息。

#### 6.2.2 功能点

- 自动生成需求摘要。
- 自动提取关键词。
- 自动生成行业、区域、主题标签。
- 生成可信度或质量评分，用于辅助人工甄别。
- 保留原始来源链接与抓取时间，方便追溯。

#### 6.2.3 业务价值

- 降低管理员阅读大量原始文本的成本。
- 提高需求分类效率。
- 为后续成果匹配与管理看板提供结构化数据基础。

### 6.3 功能模块三：成果智能匹配

#### 6.3.1 目标

围绕单条需求，自动从成果库中筛选出最相关的成果，并给出可解释的推荐结果。

#### 6.3.2 功能点

- 根据需求标题、摘要、关键词、标签与成果信息进行匹配。
- 输出匹配分数，便于结果排序和人工筛选。
- 输出匹配理由，说明推荐依据。
- 输出证据片段，帮助用户快速判断相关性。
- 支持人工触发重新匹配。

#### 6.3.3 业务价值

- 缩短从“发现需求”到“定位潜在成果”的时间。
- 提高需求与成果对接的标准化程度。
- 为成果转化线索挖掘提供统一入口。

### 6.4 功能模块四：运营闭环

#### 6.4.1 目标

让 AI 推荐结果进入真实业务流程，而非停留在展示层。

#### 6.4.2 功能点

- 支持查看需求处理状态，如未匹配、已匹配、跟进中。
- 支持人工确认匹配结果。
- 支持记录后续跟进状态和处理结果。
- 支持对重匹配、人工确认等关键动作留痕。

#### 6.4.3 业务价值

- 让需求匹配结果具备可运营性。
- 形成从“AI 推荐”到“人工推进”的闭环链路。
- 为后续评估匹配质量和运营效果积累数据。

### 6.5 一期业务流程

```mermaid
flowchart LR
    A["外部需求来源"] --> B["需求采集入池"]
    B --> C["摘要/标签/关键词生成"]
    C --> D["成果智能匹配"]
    D --> E["管理员查看匹配结果"]
    E --> F["人工确认/重新匹配"]
    F --> G["进入跟进状态"]
```

## 7. 配套能力设计

### 7.1 配套能力一：研究洞察

#### 7.1.1 定位

研究洞察不是一期主链路，但应作为管理层增值能力纳入专题范围，为甲方展示 AI 赋能后的管理价值。

#### 7.1.2 功能方向

- 按时间范围展示关键词热度变化。
- 展示关键词共现关系和交叉领域关联。
- 支持筛选近 1 年、近 3 年、全部等分析口径。
- 支持面向管理层的热点研究方向识别。

#### 7.1.3 价值说明

- 帮助管理层识别当前研究热点和交叉方向。
- 为学科布局、资源配置、重点方向研判提供辅助依据。
- 与需求洞察形成互补，兼顾“转化机会”和“方向洞察”。

### 7.2 配套能力二：成果录入助手

#### 7.2.1 定位

成果录入助手用于提升科研人员录入效率和元数据完整度，是 AI 专题中更容易快速见效的辅助能力。

#### 7.2.2 当前基础

系统当前已具备期刊等级补全能力：

- 提供 `/results/auto-fill` 接口。
- 当前真实支持的补全类型为 `journalRank`。
- 能力范围主要为期刊等级查询与结果回填。

#### 7.2.3 后续扩展方向

- DOI 元数据补全。
- 标题检索补全。
- 作者、摘要、关键词等字段自动回填。
- 自动标记补全来源，并保留人工修改权。

#### 7.2.4 价值说明

- 降低科研人员录入负担。
- 提高成果数据完整性和规范性。
- 为后续检索、统计、匹配提供更高质量的基础数据。

## 8. 分期路线图

### 8.1 一期建设范围

一期以“需求匹配闭环上线”为目标，建设范围如下：

| 模块 | 一期范围 |
| :--- | :--- |
| 需求采集管理 | 数据源配置、抓取频率、状态管理、基础去重 |
| 需求结构化处理 | 摘要、关键词、标签、可信度 |
| 成果智能匹配 | 匹配结果、匹配分、理由、证据片段、重匹配 |
| 运营闭环 | 状态管理、人工确认、跟进留痕 |
| 录入助手 | 保持现有期刊等级补全，作为专题内已落地基础能力展示 |
| 研究洞察 | 保留原型展示口径，不作为一期核心交付范围 |

### 8.2 二期建设范围

二期以“能力增强和管理看板完善”为目标，扩展范围如下：

| 模块 | 二期范围 |
| :--- | :--- |
| 研究洞察 | 热点趋势、关键词共现、交叉方向分析、管理层图谱增强 |
| 成果录入助手 | DOI/标题元数据补全、来源标记、差异确认 |
| 指标看板 | 增加需求处理效率、匹配命中率、转化线索数量等指标 |
| 运营分析 | 沉淀匹配效果回看与人工纠偏数据，为后续优化提供依据 |

### 8.3 分期策略说明

- 一期目标是“把一条业务链路跑通”，不是把所有 AI 能力一次做全。
- 二期目标是“让已跑通的链路更智能、更可视、更可运营”。
- 分期建设有利于甲方控制风险，也有利于逐步沉淀可用于优化的业务数据。

## 10. 业务价值与验收方式

### 10.1 业务价值

本专题不以“用了什么模型”作为对外核心表述，而以甲方能理解的业务价值作为主表述：

#### 价值一：提升需求处理效率

- 从多源需求人工浏览，转为系统统一采集与摘要归纳。
- 管理员可先看结构化摘要，再决定是否深入处理。

#### 价值二：提升成果转化线索发现效率

- 基于需求快速定位潜在成果。
- 缩短“发现需求”到“找到可对接成果”的时间。

#### 价值三：增强管理层的方向洞察能力

- 通过关键词热度、共现关系等方式识别重点方向。
- 辅助判断哪些研究方向值得持续投入或重点推动。

#### 价值四：提升成果基础数据质量

- 通过录入助手降低人工录入成本。
- 提高摘要、关键词、来源信息等基础元数据的完整度。

### 10.2 业务验收场景

本专题的验收应采用业务场景方式，而非单纯技术接口验收。

| 场景 | 验收描述 |
| :--- | :--- |
| 场景 1 | 管理员查看新抓取的企业需求，系统自动生成摘要、关键词和标签 |
| 场景 2 | 管理员打开某条需求详情，查看推荐成果、匹配分和匹配理由，并可手动重匹配 |
| 场景 3 | 管理层按时间范围查看热点关键词图谱，识别重点研究方向与交叉趋势 |
| 场景 4 | 科研人员录入论文时，输入期刊名称可自动补全期刊等级；后续支持 DOI/标题元数据补全 |
| 场景 5 | 当外部数据源异常或 AI 结果不可信时，系统给出提示但不阻塞人工流程 |

### 10.3 建议验收指标

建议优先采用业务可感知指标：

- 需求处理效率提升。
- 有效匹配线索数量提升。
- 管理层获取热点洞察所需时间缩短。
- 成果录入完整度提升。

项目实施阶段可根据甲方管理方式，将上述指标进一步量化为月度或季度口径。

## 11. 风险与边界

### 11.1 建设边界

一期明确不追求以下目标：

- 不追求全自动决策。
- 不追求一次性覆盖所有外部需求来源。
- 不追求把研究洞察、需求匹配、录入助手全部做到同等深度。
- 不追求在一期中完成完整的成果转化业务流程闭环。

一期坚持“AI 推荐 + 人工确认”的基本原则。

### 11.2 主要风险

#### 风险一：需求数据质量参差不齐

外部来源的需求内容可能存在噪声、重复、描述模糊等问题，会影响摘要和匹配质量。

#### 风险二：匹配结果存在误差

成果与需求的语义相关性具有一定不确定性，必须保留人工复核机制。

#### 风险三：原型能力与真实能力认知不一致

如果不明确区分前端原型与后端已实现能力，容易导致甲方对现状形成偏高预期。

#### 风险四：没有运营闭环，AI 价值难以沉淀

若只做推荐展示，不做人工确认与后续状态管理，系统难以形成真实业务价值和持续优化基础。

### 11.3 风险控制建议

- 把一期范围收敛到需求匹配闭环。
- 明确哪些能力是原型，哪些能力是一期建设项。
- 在所有 AI 输出位置保留人工确认入口。
- 用状态管理和操作留痕保证结果可追踪。

## 12. 实施建议

### 12.1 项目推进建议

建议按照“现状澄清 -> 一期闭环 -> 二期扩展”的方式推进：

1. 先与甲方完成专题方案对齐，统一对现状和目标的理解。
2. 以一期需求匹配闭环为核心，确定交付边界和验收口径。
3. 在一期运行稳定后，再扩展研究洞察和录入助手能力。

### 12.2 沟通建议

对甲方汇报时建议采用以下表述策略：

- 先讲业务问题，再讲 AI 方案，不先讲模型名称。
- 先讲一期能落地什么，再讲二期能增强什么。
- 明确说明“当前已有原型入口，但真实闭环将在后续迭代中完成”。

## 14. 最终建议

综合当前系统现状、甲方业务诉求和落地风险，建议本专题采用以下实施口径：

- 方案主线：需求匹配主线，研究洞察增值。
- 一期目标：打通需求匹配闭环，形成可上线、可验收的业务能力。
- 二期目标：增强研究洞察和成果录入助手，形成更完整的 AI 赋能体系。
- 沟通原则：明确区分“当前已实现”与“规划新增”，以免产生交付认知偏差。

该路径既能体现 AI 对成果转化与科研管理的现实价值，也能为后续更深入的智能化建设保留清晰扩展空间。
