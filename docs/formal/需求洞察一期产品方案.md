### 1.1 编写目的

本文件用于明确“需求洞察”模块一期的产品定位、范围边界、数据源策略、页面能力、状态流转、匹配逻辑和验收口径，避免项目讨论停留在“做一个爬虫功能”的层面。

需求洞察一期的本质不是建设一个泛化的爬虫平台，而是建设一套围绕成果转化线索挖掘的业务闭环系统：

**需求采集 -> 需求入池 -> 智能摘要/标签 -> 成果匹配 -> 人工确认 -> 跟进沉淀**

### 1.2 一期结论

需求洞察一期建议采用以下建设策略：

- 数据源策略上，采用白名单模式，不做全网抓取。
- 业务目标上，优先解决“需求线索发现慢、成果匹配效率低、管理缺少统一入口”三个问题。
- 产品方案上，以“系统设置中的数据源管理 + 需求洞察中的需求池与匹配详情”作为主界面承载。
- 算法策略上，采用“规则粗筛 + 智能匹配排序 + 人工确认”的组合模式。
- 实施策略上，一期只做少量高质量来源的稳定闭环，不追求大而全。

## 2. 业务背景与问题定义

### 2.1 当前业务背景

当前科研成果管理系统已经具备成果归集、审核、检索和基础统计能力，但仍以“成果被动存档”为主。随着管理诉求从“管成果”逐步转向“用成果、转成果、找机会”，系统缺少主动发现外部需求并反向连接成果库的能力。

对于高校和科研院所而言，现实业务中存在大量有价值的外部需求线索，例如：

- 政府采购和科技需求公告
- 园区、产业平台发布的合作需求
- 企业官网或公开平台发布的技术痛点
- 行业协会、专题平台发布的项目征集或技术征询

这些信息目前往往分散在不同站点，主要依赖管理员人工浏览、复制、筛选和判断，导致需求发现效率低、跟进时效差、成果转化机会流失。

### 2.2 当前核心问题

需求洞察模块要解决的不是单一技术问题，而是一组连续的业务问题：

#### 问题一：线索分散

- 需求分布在多个平台，缺少统一入口。
- 人工收集过程重复且不可追踪。
- 不同人员关注不同来源，线索积累不稳定。

#### 问题二：需求信息噪声高

- 需求原文格式不统一。
- 同一需求可能被多处重复发布。
- 需求内容有时偏长、模糊、不完整。

#### 问题三：成果匹配效率低

- 现有成果库虽然已积累数据，但缺少主动匹配场景。
- 匹配主要依赖人工经验，耗时且一致性差。
- 缺乏统一的匹配分、匹配理由和证据支撑。

#### 问题四：缺少运营闭环

- 即使发现了需求，也可能没有后续状态管理。
- 无法沉淀哪些需求有效、哪些匹配命中、哪些最终被跟进。
- 价值难以量化，系统难以持续优化。

## 3. 产品目标与定位

### 3.1 产品目标

需求洞察一期的产品目标可以归纳为三点：

1. 建立统一的外部需求线索采集入口。
2. 建立需求与成果库之间的标准化匹配链路。
3. 建立“AI 推荐 + 人工确认”的可运营闭环。

### 3.2 产品定位

需求洞察一期不定义为“爬虫系统”，而定义为：

**面向科研成果转化场景的需求线索采集与智能匹配系统。**

其中：

- `爬虫` 是采集手段；
- `需求池` 是业务载体；
- `智能匹配` 是核心价值；
- `人工确认与跟进` 是闭环落点。

### 3.3 一期成功标准

若一期要被定义为成功，至少应满足以下条件：

- 可以稳定接入少量高质量数据源。
- 可以把原始需求转成结构化需求对象。
- 可以针对单条需求返回可解释的成果匹配结果。
- 可以支持管理员进行人工确认、重匹配和跟进状态维护。
- 可以通过若干核心指标看出线索沉淀和匹配效果。

## 4. 一期建设范围与范围边界

### 4.1 一期建设范围

一期聚焦以下 4 个模块：

#### 模块一：数据源管理

- 配置数据源名称、类型、URL、认证方式、抓取频率、优先级、启停状态。
- 记录最近运行时间、最近成功时间、失败原因、健康状态。

#### 模块二：需求入池与结构化

- 抓取需求原文并入池。
- 自动生成摘要、关键词、行业标签、地域标签。
- 执行去重和基础质量评估。
- 形成统一的需求对象用于展示和匹配。

#### 模块三：成果智能匹配

- 将需求与成果库进行关联匹配。
- 输出匹配分、匹配理由、证据片段。
- 支持人工触发重匹配。

#### 模块四：人工运营闭环

- 维护需求状态。
- 支持人工确认有效需求与有效匹配。
- 支持进入跟进状态并沉淀结果。

### 4.2 一期明确不做

为了控制范围，一期明确不做以下内容：

- 不做全网抓取能力。
- 不做登录态复杂、反爬严格或授权不明确的数据源。
- 不做自动生成完整转化流程。
- 不做无人值守的全自动决策。
- 不把研究洞察图谱作为一期核心交付内容。

### 4.3 一期对外口径

需求洞察一期对外应表述为：

**先做白名单数据源下的高质量需求线索采集与成果匹配闭环。**

不建议表述为：

- 已支持全网需求自动发现。
- 已具备完全自动化转化推荐能力。
- 已形成无需人工介入的 AI 决策系统。

## 5. 数据源策略

### 5.1 数据源选择原则

一期的数据源策略是成败关键，选择原则如下：

- 来源公开，抓取合规风险低。
- 页面结构或接口相对稳定。
- 内容与科研成果转化相关性高。
- 站点更新频率适中，具有持续线索价值。
- 能够被抽象为标准需求对象，而不是纯新闻内容。

### 5.2 一期数据源分级

建议将数据源划分为三类：

#### A 类：优先接入

适合一期优先接入的数据源：

- 政府采购类公开站点
- 科技厅/科技局需求发布平台
- 园区、产业平台公开需求页
- 合作单位公开项目征集页
- 有 RSS 或开放接口的专题平台

特征：

- 公开、稳定、可信度高
- 内容结构相对规范
- 与业务目标直接相关

#### B 类：择优接入

- 行业协会或专题站点
- 企业官网公开需求栏目
- 区域合作平台的需求公告页

特征：

- 价值可能较高
- 但结构不一定稳定，噪声略高

#### C 类：一期不建议接入

- 泛资讯新闻站
- 强登录态平台
- 强反爬平台
- 用户生成内容社区
- 无明确需求字段、以新闻报道为主的站点

特征：

- 抓取成本高
- 噪声大
- 有效需求占比低

### 5.3 一期数据源数量建议

一期建议控制在 `3-5 个高质量来源`，不建议超过 8 个。

原因如下：

- 便于快速验证价值。
- 便于管理员建立使用习惯。
- 便于团队聚焦处理结构化和匹配质量，而不是陷入数据源维护。

### 5.4 数据源管理能力

数据源管理页面建议包含以下字段：

| 字段 | 说明 |
| :--- | :--- |
| 名称 | 数据源名称 |
| 类型 | HTML、RSS、API、文件等 |
| 行业 | 主要行业标签 |
| 地域 | 主要区域标签 |
| 基础 URL | 抓取入口地址 |
| 认证方式 | 无认证、API Key、Cookie、Basic Auth |
| 抓取频率 | 以小时为单位 |
| 优先级 | 高、中、低 |
| 启用状态 | 是否启用 |
| 最近运行时间 | 最近一次任务执行时间 |
| 最近成功时间 | 最近一次成功抓取时间 |
| 健康状态 | healthy、warning、error、idle |
| 失败原因 | 最近一次异常说明 |

## 6. 核心业务流程

### 6.1 一期业务流程

```mermaid
flowchart LR
    A["白名单数据源"] --> B["需求抓取"]
    B --> C["去重与清洗"]
    C --> D["摘要/关键词/标签生成"]
    D --> E["进入需求池"]
    E --> F["成果库匹配"]
    F --> G["管理员查看详情"]
    G --> H["人工确认 / 重匹配"]
    H --> I["进入跟进状态"]
```

### 6.2 产品状态流

需求对象建议采用如下状态：

| 状态 | 说明 |
| :--- | :--- |
| `new` | 新入池，尚未研判 |
| `reviewing` | 已被管理员查看或研判中 |
| `matched` | 已产生可参考的匹配结果 |
| `in_follow_up` | 已被人工确认并进入跟进 |
| `invalid` | 判定为无效需求或低价值需求 |
| `archived` | 已过期或处理完成归档 |

说明：

- `matched` 不代表一定要跟进，只代表系统已给出可参考结果。
- `in_follow_up` 必须是人工确认后进入。
- `invalid` 用于沉淀噪声需求，避免反复出现。

### 6.3 关键动作定义

| 动作 | 执行人 | 结果 |
| :--- | :--- | :--- |
| 抓取 | 系统 | 生成原始需求记录 |
| 结构化处理 | 系统 | 生成摘要、标签、关键词、可信度 |
| 自动匹配 | 系统 | 生成候选成果列表 |
| 查看匹配 | 管理员 | 进入人工研判 |
| 重新匹配 | 管理员 | 刷新匹配结果 |
| 确认有效 | 管理员 | 标记为有效需求 |
| 跟进 | 管理员/业务负责人 | 进入跟进状态 |
| 标记无效 | 管理员 | 从有效运营列表中移除 |

## 9. 匹配策略建议

### 9.1 一期匹配原则

一期不建议使用“纯黑盒大模型直接给最终结果”的模式，而建议使用组合式策略：

**规则粗筛 + 语义匹配排序 + 人工确认**

### 9.2 规则粗筛

优先使用结构化条件先缩小成果候选集，例如：

- 行业标签匹配
- 地域标签辅助筛选
- 关键词重叠
- 成果类型限制
- 时间范围限制

作用：

- 降低语义匹配范围
- 提高结果稳定性
- 减少无关成果进入候选集

### 9.3 智能排序

在规则粗筛后的候选结果中，再进行更细粒度的相关性排序。

推荐输出：

- 匹配分
- 匹配理由
- 证据片段

作用：

- 让管理员能理解“为什么推荐这条成果”
- 提高推荐的可解释性

### 9.4 人工确认

人工确认是一期不可省略的步骤，原因如下：

- 需求文本具有模糊性。
- 成果文本质量可能参差不齐。
- 是否真正值得跟进，仍依赖业务判断。

因此，一期匹配结果只能定义为“推荐候选”，不能定义为“自动决策”。

## 11. 验收场景与指标

### 11.1 核心验收场景

#### 场景一：管理员管理数据源

- 可以新增白名单数据源
- 可以查看运行状态
- 可以测试连接
- 可以启停采集

#### 场景二：系统自动产生需求池

- 系统能够从白名单数据源抓取需求
- 能对重复需求做基本识别
- 能展示结构化字段

#### 场景三：管理员查看需求详情

- 能看到摘要、关键词、标签、来源、可信度
- 能看到候选成果和匹配理由

#### 场景四：管理员进行人工研判

- 能重新匹配
- 能标记有效或无效
- 能把需求推进到跟进状态

#### 场景五：管理者查看成效

- 能看到抓取数量、有效数量、匹配数量、跟进数量等指标

### 11.2 建议指标

一期建议采用以下指标衡量成效：

| 指标 | 含义 |
| :--- | :--- |
| 抓取需求总数 | 系统从数据源采集到的总量 |
| 去重后有效需求数 | 剔除重复和低质后的需求数 |
| 可匹配率 | 有候选成果的需求占比 |
| 人工确认有效率 | 被管理员确认有效的需求占比 |
| 跟进率 | 进入跟进状态的需求占比 |
| 数据源健康率 | 正常运行的数据源占比 |

### 11.3 指标解释原则

指标用于反映系统的业务价值，不建议在一期阶段过早追求复杂模型指标。对甲方汇报时，优先讲：

- 管理效率是否提升
- 有效线索是否增多
- 跟进动作是否变快
- 数据源是否稳定

---

## 12. 风险与建议

### 12.1 主要风险

#### 风险一：把“爬虫”当成产品主体

如果团队把重点放在“抓更多网站”，而不是“沉淀有效需求线索”，很容易造成系统复杂但价值不清。

#### 风险二：数据源过多导致维护失控

数据源越多，页面变化、反爬、失效维护成本越高。一期不适合贪多。

#### 风险三：成果库元数据质量不足

如果成果本身的标题、摘要、关键词不完整，匹配效果会明显下降。

#### 风险四：没有人工闭环

如果没有状态管理和人工确认，需求洞察会退化成一个“看起来很智能的列表页”。

### 12.2 风险控制建议

- 一期仅接入少量高质量白名单来源。
- 必须设计状态机和人工动作。
- 匹配结果必须带理由和证据。
- 与成果录入质量提升工作并行推进。

---

## 13. 最终产品建议

需求洞察一期建议按以下口径推进：

- 产品名称：需求洞察
- 核心定位：需求线索采集与成果智能匹配
- 一期目标：白名单数据源下打通稳定闭环
- 一期重点：不是抓得多，而是抓得准、匹得准、能运营
- 交付重点：数据源管理、需求池、需求详情、匹配解释、人工跟进

一句话总结：

**需求洞察一期要做的，不是“全网爬虫”，而是“面向成果转化的高质量需求线索运营系统”。**
