# 需求洞察与研究洞察产品边界及信息架构方案

## 1. 文档说明

| 项目 | 内容 |
| :--- | :--- |
| 文档名称 | 需求洞察与研究洞察产品边界及信息架构方案 |
| 版本 | v0.1 |
| 日期 | 2026-04-14 |
| 面向对象 | 甲方业务负责人、项目负责人、产品经理、实施团队、研发团队 |
| 文档定位 | 用于统一“需求洞察”与“研究洞察”的产品边界、页面职责、菜单结构与验收口径 |
| 文档原则 | 明确边界、减少混用、支撑一期落地、为后续研发实施提供统一口径 |

### 1.1 编写目的

当前系统中，“需求洞察”和“研究洞察”都已具备一定前端原型基础，但两者在业务目标、分析对象、主流程与价值表达上存在天然差异。

如果继续以模糊方式推进，容易出现以下问题：

- 将“需求洞察”做成资讯列表或泛化爬虫平台，缺少运营闭环。
- 将“研究洞察”做成展示型词云页，缺少管理决策价值。
- 页面、菜单、指标和接口命名混用，导致甲方、产品、研发对模块定位理解不一致。

本文件用于明确以下问题：

- `需求洞察` 与 `研究洞察` 分别解决什么问题。
- 两个模块分别面向哪些用户、使用哪些数据、产出哪些结论。
- 一期页面结构、菜单入口、角色权限和联动关系应如何定义。
- 一期应如何设置指标与验收口径，避免后续实现偏离目标。

### 1.2 文档结论

本次方案建议明确采用以下策略：

- `需求洞察` 定义为面向成果转化场景的外部机会运营系统。
- `研究洞察` 定义为面向管理层的内部方向决策系统。
- 两个模块共享底层主题词典与关键词标准化能力，但不共享主页面、不共享主流程、不共享主指标。
- 菜单层建议统一归入 `智能洞察` 父级，以体现“同一能力域下的两条业务线”。
- 一期保持“需求洞察主线、研究洞察增值”的策略，优先保证需求运营闭环可落地，同时把研究洞察建设为管理层可使用的分析入口。

## 2. 背景与问题定义

### 2.1 当前现状

结合当前仓库现状，系统已经具备如下基础：

- 需求洞察前端页面已具备 `需求列表 + 需求详情 + 匹配结果 + 重匹配` 的基础壳。
- 研究洞察前端页面已具备 `关键词图谱 + 时间筛选` 的基础展示壳。
- Mock 数据层中已存在需求、匹配结果、数据源、抓取配置等样例数据。
- AI 专题 PRD 已提出“需求洞察主线、研究洞察增值”的总体方向。

但同时也存在明显问题：

- 研究洞察当前过于依赖图谱展示，尚未形成管理层可直接使用的方向结论。
- 需求洞察当前仍偏重列表与详情，跟进看板、数据源管理、运营指标承载不足。
- 菜单层虽然将两者并列展示，但缺少对“运营线”和“决策线”的清晰区分。
- 两边都使用“关键词”“热点”等概念，但语义不同，存在混淆风险。

### 2.2 为什么必须拆开定义

`需求洞察` 和 `研究洞察` 虽然都属于智能分析能力，但它们并不是同一个模块的两个页面，而是两类不同的产品问题：

- 需求洞察要回答的是：`外部有哪些真实需求值得跟进，哪些需求与现有成果匹配，哪些线索正在推进。`
- 研究洞察要回答的是：`内部在哪些研究主题有积累，外部哪些方向正在升温，哪些方向值得加码或组织交叉。`

如果不拆分，后续将同时出现两种偏差：

- 把运营流程问题误做成可视化分析问题。
- 把方向决策问题误做成线索处理问题。

因此，一期必须先完成产品定义上的拆分，再进入实现层设计。

## 3. 模块定位与边界

### 3.1 一句话定义

- `需求洞察`：面向成果转化场景的外部需求线索采集、匹配与跟进系统。
- `研究洞察`：面向管理层的研究主题识别、趋势研判与方向决策系统。

### 3.2 模块对比表

| 维度 | 需求洞察 | 研究洞察 |
| :--- | :--- | :--- |
| 核心问题 | 外部有哪些需求值得跟进 | 内部哪些方向值得布局 |
| 服务对象 | 科研管理员、转化运营人员、科研秘书 | 管理层、院领导、科研管理负责人 |
| 分析对象 | 企业需求、采购公告、合作需求、政策征集 | 成果、论文、专利、主题、关键词、外部文献趋势 |
| 主数据源 | 外部需求源为主，内部成果库为匹配底座 | 内部成果库为主，外部文献为趋势参照 |
| 主动作 | 采集、去重、摘要、标签、匹配、确认、跟进 | 归类、聚合、趋势、交叉、解释、建议 |
| 主输出 | 需求池、匹配结果、跟进状态、运营指标 | 热点主题、强势方向、交叉方向、机会缺口 |
| 决策类型 | 业务执行决策 | 管理配置决策 |
| 更新节奏 | 高频、日常运营 | 中频、月度/季度研判 |

### 3.3 边界约束

为保证模块职责清晰，一期需明确以下边界：

#### 需求洞察不负责

- 机构学科布局判断
- 研究方向长期演化结论
- 管理层资源配置建议
- 基于论文/专利主题的方向图谱研判

#### 研究洞察不负责

- 需求线索的人工跟进管理
- 需求状态流转与推进
- 数据源抓取配置与异常运维
- 单条需求的成果匹配详情

### 3.4 一期总体策略

一期应保留以下总体策略：

- `需求洞察` 作为主线模块，优先形成可用的运营闭环。
- `研究洞察` 作为增值模块，优先形成可用的管理视图。
- 两者可联动，但不强行合并为一个大而全模块。

## 4. 页面职责与信息架构

### 4.1 顶层菜单建议

当前菜单建议从 `科技助手` 统一调整为：

**智能洞察**

其下固定包含两个子模块：

- `需求洞察`
- `研究洞察`

这样既保留两者的归属关系，又不会把它们误解为一个页面下的两个功能点。

### 4.2 角色入口建议

| 角色 | 默认关注模块 | 默认入口 |
| :--- | :--- | :--- |
| 管理员 | 需求洞察 + 研究洞察 | 需求洞察需求池 / 管理看板 |
| 科研秘书 / 转化运营 | 需求洞察 | 需求池列表 |
| 管理层 / 决策者 | 研究洞察 | 研究概览 |
| 普通科研人员 | 需求洞察（浏览级） | 需求池列表 |

### 4.3 需求洞察页面结构

需求洞察一期建议固定为 4 个页面或等价 4 个稳定信息区域：

#### 页面一：需求池列表

核心职责：

- 统一查看新入池需求
- 根据条件快速筛选高价值需求
- 识别高可信、高匹配、高优先级需求

核心字段建议：

- 需求标题
- 来源站点 / 来源类型
- 行业 / 地域
- 抓取时间
- 可信度
- 当前最佳匹配分
- 当前状态

#### 页面二：需求详情与匹配

核心职责：

- 查看单条需求的结构化内容
- 查看候选成果、匹配理由和证据片段
- 对需求进行重新匹配与人工判断

核心内容建议：

- 原始摘要
- 智能摘要
- 关键词与标签
- 来源链接
- 候选成果列表
- 匹配分 / 匹配理由 / 证据片段
- 跟进动作入口

#### 页面三：跟进看板

核心职责：

- 看有效需求沉淀情况
- 看需求状态流转情况
- 看待处理、已确认、跟进中、超期未处理线索

核心指标建议：

- 新入池需求数
- 人工确认有效需求数
- 跟进中需求数
- 超期未处理需求数
- 已形成转化线索需求数

#### 页面四：数据源管理

核心职责：

- 管理数据源白名单
- 查看抓取频率、健康状态、失败原因
- 调整去重阈值、通知方式等运营参数

核心字段建议：

- 数据源名称
- 来源类型
- 行业 / 地域
- 基础地址
- 抓取频率
- 启停状态
- 最近运行时间
- 最近成功时间
- 健康状态
- 失败原因

### 4.4 研究洞察页面结构

研究洞察一期建议固定为 4 个页面或等价 4 个稳定信息区域：

#### 页面一：研究概览

核心职责：

- 先给管理层结论，而不是先给图
- 快速回答“哪些主题升温、哪些方向强、哪些方向值得投入”

核心模块建议：

- 新兴主题 Top N
- 强势主题 Top N
- 交叉主题 Top N
- 高潜缺口主题 Top N
- 重点主题近 12 个月增速
- 多部门参与主题占比

#### 页面二：主题池

核心职责：

- 以主题为单位查看趋势、内部基础、外部热度和建议动作
- 支持按主题类型、学科、部门、时间范围筛选

主题卡核心字段建议：

- 主题名称
- 所属学科域
- 外部热度
- 内部基础
- 近 12 个月增速
- 参与部门数
- 建议动作

#### 页面三：主题详情

核心职责：

- 查看单主题的时间序列、内部代表成果、外部代表文献与解释结论

核心内容建议：

- 主题定义
- 时间序列趋势
- 内部代表成果
- 外部代表文献
- 参与部门分布
- 建议动作与解释

#### 页面四：关系网络

核心职责：

- 作为证据层与探索层，辅助查看主题共现与交叉关系

设计原则：

- 不作为研究洞察首页
- 不直接承载管理结论
- 主要用于验证“交叉方向”“主题关联”等判断

### 4.5 当前前端原型的调整建议

结合当前仓库现状，一期建议做如下产品层调整：

- 当前 `需求洞察` 页面保持“需求池 + 抽屉详情”逻辑，但补齐跟进看板和数据源管理承载。
- 当前 `研究洞察` 页面不再以“关键词图谱”作为首页主体，而应重构为“研究概览”。
- 原有关键词关系图保留，但应下沉为研究洞察中的“关系网络”子视图。

## 6. 联动规则

### 6.1 研究洞察到需求洞察

一期建议只保留一条明确联动：

- 管理层可将重点主题加入 `观察方向`。
- 需求洞察对相关方向的外部需求做优先标记和优先展示。

这条联动用于体现“研究洞察定方向，需求洞察找机会”。

### 6.2 需求洞察到研究洞察

一期建议只保留一条弱联动：

- 外部需求热度可回流为 `方向关注信号`。
- 该信号只作为管理层参考，不直接改写研究主题热度分。

### 6.3 概念口径区分

#### 关于关键词

- 需求洞察中的关键词：用于需求内容召回、匹配和筛选。
- 研究洞察中的关键词：用于主题聚类、趋势解释和关系网络。

#### 关于热点

- 需求洞察中的热点：市场、企业、政策、园区等外部需求热点。
- 研究洞察中的热点：内部成果与外部文献共同映射下的科研主题热点。

## 7. 指标口径

### 7.1 需求洞察核心 KPI

一期建议固定为：

- 新入池需求数
- 有效需求数
- 有匹配结果占比
- 人工确认有效率
- 跟进中线索数
- 数据源健康度

### 7.2 研究洞察核心 KPI

一期建议固定为：

- 新兴主题数
- 强势主题数
- 交叉主题数
- 高潜缺口主题数
- 重点主题近 12 个月增速
- 多部门参与主题占比

### 7.3 指标使用原则

一期必须明确以下原则：

- 禁止将 `需求数量` 直接解释为 `研究实力`。
- 禁止将 `论文/专利数量` 直接解释为 `外部机会`。
- 研究洞察中的分数只用于方向判断，不用于个人绩效排名。
- 指标必须服务具体业务问题，而不是作为脱离语境的总分展示。

## 9. 一期验收口径

### 9.1 需求洞察验收场景

当用户进入需求洞察后，应能在 3 步内回答：

1. 这条需求值不值得跟。
2. 它匹配哪些成果。
3. 当前谁在跟进、下一步做什么。

若页面只能看到需求列表，但无法支持人工确认与跟进，则不能视为一期完成。

### 9.2 研究洞察验收场景

当用户进入研究洞察后，应能在 3 步内回答：

1. 当前哪些主题在升温。
2. 我们在哪些主题已有基础。
3. 哪些方向值得投入或组织交叉。

若页面只有图谱和词云，不能明确给出方向判断与建议动作，则不能视为一期完成。

### 9.3 整体验收原则

- 任何一张页面都不能同时承载“需求状态流转”和“机构方向结论”两套主叙事。
- 一个指标只能属于一个模块，不能既作为运营指标又作为战略指标。
- 菜单命名、页面标题、接口命名应让业务方和研发一眼分清“机会运营”与“方向决策”。

## 10. 与现有仓库的对齐建议

结合当前仓库实现，后续研发落地时建议按以下口径推进：

- 当前需求洞察主页面继续以 `research-management-system/src/views/insights/DemandInsights.vue` 为主承载。
- 当前研究洞察主页面应以 `research-management-system/src/views/admin/ResearchInsights.vue` 为重构入口。
- 当前菜单配置位于 `research-management-system/src/layouts/MainLayout.vue`，需在后续实现阶段完成“科技助手 -> 智能洞察”的调整。
- 当前需求接口定义位于 `research-management-system/src/api/demand.ts`，研究洞察后续建议从 `result.ts` 中拆分独立接口域。
- 当前 mock 数据中已有需求、匹配、数据源、抓取设置等基础样例，可优先支撑一期前端结构重构。
